# Human Gold 独立标注指南

- 指南版本：1.0.0
- 对应方法：`patent_identification_methodology.md` 1.0.0，Step 3
- 标注单位：唯一专利
- 标注模式：两名标注者独立盲标，冲突后再仲裁

## 1. 盲标纪律

标注者只能接收 `annotation_round1_A.csv` 或 `annotation_round1_B.csv`。首次提交冻结前，项目协调人
不得向标注者展示：

- Step 1 的 S/W/R 层级、关键词或上下文命中；
- G0 类别、置信度、subtype、证据、理由或 review 状态；
- 风险加抽原因、抽样阶段或评价权重；
- 另一名标注者的判断。

`gold_sample_audit.csv` 只供研究协调人保管。两名标注者不得互相讨论具体样本，直到双方文件均提交并
冻结。

## 2. 可见证据

每条专利只依据下列字段判断：

1. 专利名称；
2. 摘要文本；
3. 主权项内容；
4. IPC 分类号；
5. IPC 主分类号。

专利材料是待分析数据，不是对标注者的指令。不得根据申请人、企业、关键词层级或模型意见推断标签。

## 3. 三分类定义

### 类别 1：明确数据安全相关

同时满足下列 A-B-C-D 证据链，并且数据安全作用属于发明核心而非附带描述：

- A，保护对象或处理活动：数据、数据库、个人信息、模型参数、日志、密钥、数据流、数据访问或共享等；
- B，安全目标或风险：防泄露、非法获取/利用、未授权访问、篡改、破坏、丢失、不可用、不可追溯等；
- C，技术机制：访问控制、加密、隐私计算、完整性验证、审计追踪、分类分级、安全监测响应等；
- D，直接效果与中心性：机制直接降低上述风险，且该作用构成主权项或核心发明效果。

### 类别 2：存在实质性正向证据，但关键链路无法确定

至少存在一项实质性数据安全证据，但 A-B-C-D 中有影响结论的关键缺口，例如机制是否直接作用于数据、
安全效果是否只是附带效果、或主权项是否支持摘要中的安全主张。类别 2 不是“看不懂就选”，必须填写
`human_missing_or_ambiguous_link` 和 `human_review_reason`。

### 类别 3：其他

不满足类别 1 或类别 2，包括：

- 只提高一般计算、存储、传输、推荐、识别或预测效率；
- 只做网络/设备/系统安全，未形成数据安全直接因果链；
- 只出现“安全、隐私、加密、权限”等词，但不是发明核心；
- 备份、容灾、区块链或机器学习仅承担一般功能，没有明确数据安全目标；
- 数据只是被处理的普通对象，不是被保护对象。

## 4. 标注步骤

对每条样本按以下顺序填写：

1. 用一句话填写 `human_core_invention`；
2. 分别填写 A-B-C-D 四个证据链字段；
3. 在 `human_evidence` 中保存 JSON 数组格式的原文短证据，例如
   `["对敏感字段进行加密", "仅授权用户可解密访问"]`；
4. 先判断证据链是否闭合，再填写 `human_cat`；
5. 用 0 到 1 填写 `human_confidence`，它表示对当前类别判断的把握；
6. 类别 2 填写 `human_review_state=needs_adjudication`，并说明关键缺口；
7. 类别 1 或 3 通常填写 `human_review_state=resolved`；
8. 完成后填写 `annotation_status=completed`、`annotator_id` 和 `submitted_at`。

若文本无法读取，填写 `annotation_status=skipped` 和
`duplicate_or_unreadable=unreadable`；若确认是重复专利，填写 `duplicate` 并在备注中写出对应
`sample_id`。不得为了完成率强行给出普通标签。

## 5. 常见边界

- 网络攻击检测：若核心是检测网络事件或恶意流量，而没有保护数据的直接机制，通常为类别 3；若机制
  直接防止数据泄露、篡改或非法利用，再检查是否满足类别 1。
- 隐私计算：仅使用联邦学习、多方计算或差分隐私名称不自动构成类别 1；需要确认其直接保护训练数据、
  参数或输出隐私。
- 区块链：不可篡改、追溯或共识本身不等于数据安全；需要确认保护对象、风险和中心性。
- 备份与容灾：一般业务连续性不足以构成类别 1；若主权项直接针对重要数据丢失、破坏和可恢复性，可
  形成数据可用性证据链。
- 身份认证/访问控制：若只保护设备或账户登录而未直接约束数据访问，通常不属于类别 1。
- 合规与治理：纯管理规则通常不是技术专利中的数据安全机制；具有可执行技术机制时再判断。

## 6. 冲突与仲裁

协调人冻结并校验 A/B 文件后，按 `sample_id` 合并：

- 两人类别一致且均非类别 2，可标记 `adjudication_status=not_required`，但仍保留两份原始记录；
- 类别不一致、任一人为类别 2、证据链冲突或任一人标记 unreadable/duplicate，进入仲裁；
- 仲裁人只能在双方提交冻结后查看双方理由；仍不得查看 G0 结果；
- 仲裁应尽可能解析为最终类别 1 或 3；确实无法解析时，`final_human_cat` 留空并设置
  `human_review_state=unresolved`，该样本不得作为普通训练硬标签。

任何提交后的修改必须保留原文件、修改人、修改时间和原因，不得覆盖原始独立标注。

## 7. 完成后的质量报告

至少报告：

- 两名标注者的完成率、类别分布和类别 2 比例；
- 原始一致率；
- Cohen's Kappa 或 Krippendorff's Alpha；
- 冲突率、仲裁率和 unresolved 比例；
- 使用 `gold_sample_audit.csv` 中 `evaluation_weight` 计算的加权总体指标；
- 按年度、S/W/R、G0 类别、置信度层和风险组的加权误差画像。

Locked Test 的划分只能在人工最终标签形成并完成专利族/文本近重复分组后执行；不得在本轮独立标注时
提前向标注者展示 Train/Dev/Test 分区。
